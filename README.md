# AI-Fitting — 문제 정의서 (딥러닝 기반)

## 0. 저장소 실행 요약 (FASHN VTON 로컬)

- **구성:** Next.js [`web/`](web/) + FastAPI [`backend/`](backend/). 가상 피팅은 [FASHN VTON v1.5](https://github.com/fashn-AI/fashn-vton-1.5) **사전학습 가중치**로 로컬 GPU에서 추론합니다 (Replicate 등 외부 유료 API 없음).
- **가중치:** 공식 레포의 `scripts/download_weights.py` 로 약 2GB를 받습니다. 저장소 루트에서 `bash scripts/setup-fashn-weights.sh` 를 실행하면 `backend/weights/` (또는 `FASHN_WEIGHTS_DIR`) 에 저장됩니다. `backend/weights/` 는 Git에 포함하지 않습니다.
- **학습:** FASHN 공식 README는 **엔드유저용 학습·데이터셋 학습 가이드를 제공하지 않습니다.** 본 프로젝트도 사전학습 추론만 사용합니다.

**백엔드 (Linux/WSL2 + NVIDIA 권장):**

```bash
cd backend
python -m venv .venv && source .venv/bin/activate
# PyTorch CUDA 휠: https://pytorch.org/get-started/locally/
pip install -r requirements.txt
cd .. && bash scripts/setup-fashn-weights.sh
cd backend && uvicorn main:app --reload --host 0.0.0.0 --port 8000
```

`backend/.env` 예: `FASHN_WEIGHTS_DIR=/절대경로/.../backend/weights`, `FASHN_CATEGORY=tops` (`tops` \| `bottoms` \| `one-pieces`). 기존 `REPLICATE_CATEGORY`(`upper_body` 등)도 `tops`/`bottoms`/`one-pieces` 로 매핑됩니다. `POST /api/try-on` 응답의 `result_url` 은 `data:image/png;base64,...` 입니다.

**프론트:** `cd web && npm install && npm run dev` — `NEXT_PUBLIC_API_URL` 로 백엔드 URL 지정 가능.

**GitHub 푸시:** `git remote add origin https://github.com/<user>/<repo>.git` 후 `git push -u origin main` (인증은 PAT/SSH).

본 문서는 가상 피팅(Virtual Try-On)을 딥러닝 관점에서 정의하고, 데이터·모델·지표·한계를 한곳에 정리합니다. 실험 수치는 환경에 따라 달라지므로, 아래 표와 문단은 **보고서 작성 시 실제 로그·검증 세트로 갱신**하면 됩니다.

## 1. 문제 정의서

### 1.1 해결할 문제

- **과제:** 사람(상반신/전신) 이미지와 의류(평면 또는 착용 참조) 이미지를 입력으로 받아, **의류가 자연스럽게 착용된 결과 이미지**를 생성하는 것.
- **딥러닝 관점:** 조건부 이미지 생성(conditional image synthesis) 또는 이미지-이미지 변환. 인물의 자세·조명·가림(occlusion)이 달라도 **의류의 질감·무늬·경계**가 왜곡되지 않도록 학습된 표현을 활용해야 함.
- **운영 맥락:** 웹 서비스에서 사용자 업로드 이미지에 대해 **사전학습(pretrained) 가상 피팅 모델**(예: diffusion / VTON 계열)을 호출해 결과를 반환하는 파이프라인.

### 1.2 예측 목표

| 구분 | 목표 |
|------|------|
| **정성** | 착용감이 자연스럽고, 원 의류 색·패턴이 보존된 결과 |
| **정량** | 검증 세트에서 정의한 지표(아래 4절)가 **베이스라인 대비 개선** 또는 **서비스 SLA 수준 이상** |
| **비기능** | 추론 지연·GPU 메모리·API 요청 크기 제한 내에서 안정 동작 |

## 2. 데이터 설명

### 2.1 출처 (예시)

- **공개 데이터셋:** DeepFashion, VITON-HD, DressCode 등 가상 피팅·패션 분야 벤치마크(라이선스 확인 후 사용).
- **내부/서비스 데이터:** 실제 사용자 업로드(개인정보·저작권 정책에 따라 수집·가명화 여부 명시).

### 2.2 규모

- 학습/검증/테스트 **분할 비율**과 **이미지 수(장)** 를 표로 기록한다.  
  예: Train N₁장, Val N₂장, Test N₃장 (해상도 분포, 전신/상반신 비율 포함 권장).

### 2.3 주요 변수

| 변수 | 설명 |
|------|------|
| **person / human** | 착용 대상 인물 이미지 (RGB, 해상도 H×W) |
| **garment / garm** | 합성할 의류 이미지 (마스크·세그멘테이션 유무는 데이터셋 정의에 따름) |
| **pose / parsing** (해당 시) | 자세 키포인트, 인체 파싱 맵 등 보조 입력 |
| **target** | 정답 착용 이미지(지도학습·평가용) 또는 사용자 만족도 라벨(선택) |

API 연동 시에는 모델 스키마에 맞는 필드명(`human_img`, `garm_img` 등)으로 매핑한다.

## 3. 기본 모델 실행

### 3.1 첫 번째 ML(딥러닝) 모델

- **접근:** 사전학습 가상 피팅 모델을 **추론(inference)** 으로 로컬 실행(FASHN VTON v1.5 가중치).
- **입력:** 전처리된 사람 이미지 + 의류 이미지(필요 시 리사이즈·정규화).
- **출력:** 합성 이미지(텐서 또는 PNG/JPEG).

### 3.2 실행 결과 제시 형식 (보고서에 채울 항목)

1. **환경:** GPU 모델명, 배치 크기, 프레임워크(PyTorch 등), 모델 식별자(슬러그/버전).
2. **한 장 또는 소수 샘플:** 입력 쌍과 출력 이미지를 나란히 배치한 figure.
3. **검증 세트 요약:** 평균 추론 시간(초/장), 실패율(타임아웃·검증 오류).
4. **정량 지표:** 4절의 지표를 Val/Test에 대해 **한 줄 표**로 정리.

> *수치는 실험 후 여기에 기입. 예: Test LPIPS = ··· , FID = ··· , 추론 p50 = ··· ms.*

## 4. 성능 지표 해석

가상 피팅은 **픽셀 단순 분류가 아닌 생성 과제**이므로, 분류 지표와 생성·품질 지표를 함께 쓰는 것이 일반적이다.

### 4.1 분류·검출 계열 (보조 라벨이 있을 때)

- **Accuracy:** 전체 예측 중 맞은 비율. 클래스 불균형이 있으면 **단독 사용은 위험**.
- **Precision (정밀도):** 모델이 “양성”이라고 한 것 중 실제 양성 비율. **오탐(false positive) 비용**이 클 때 중요.
- **Recall (재현율):** 실제 양성 중 모델이 잡아낸 비율. **놓침(false negative) 비용**이 클 때 중요.
- **F1-score:** Precision과 Recall의 조화평균. 둘의 균형을 한 숫자로 볼 때 유용.

### 4.2 생성·이미지 품질 (가상 피팅에 자주 사용)

- **LPIPS / SSIM:** 지각적·구조적 유사도. 정답 이미지가 있을 때 **픽셀/지각적 근접도** 참고.
- **FID:** 생성 분포와 실제 분포 거리. **세트 단위** 품질 경향 파악에 사용.
- **사람 평가:** A/B 선호도, MOS(평균 주관 점수) — **서비스 품질**과 가장 잘 맞는 경우가 많음.

보고서에는 **어떤 지표를 왜 선택했는지** 한 문단으로 명시한다.

## 5. 현재 한계 정리

### 5.1 모델·데이터 측면

- **도메인 갭:** 학습 분포와 실제 사용자 사진(조명, 배경, 자세) 차이로 **아티팩트·색 번짐** 발생 가능.
- **가림·포즈:** 팔·머리카락 등에 가려진 영역에서 **의류 경계 불안정**.
- **해상도:** 고해상도 입력은 API/메모리 제한에 걸릴 수 있어 **디테일 손실**.

### 5.2 운영·딥러닝 파이프라인 측면

- **로컬 GPU 의존:** VRAM·드라이버·CUDA 정합, 추론 지연이 서비스 SLA에 직접 영향.
- **학습 미수행 시:** 도메인 특화 **미세조정(fine-tuning)** 이 없으면 특정 의류/체형에서 성능 한계가 명확할 수 있음.

### 5.3 개선 방향

- 데이터: 도메인에 가까운 **파인튜닝용 소량 고품질 쌍** 수집, 증강·세그멘테이션 전처리 강화.
- 모델: 최신 VTON/diffusion 계열 비교, LoRA 등 **효율적 적응** 검토.
- 평가: 사람 평가 프로토콜 고정 + LPIPS/FID 등 **다지표**로 회귀 방지.
- 시스템: 입력 리사이즈 정책, 캐싱, 비동기 큐로 **지연·실패율** 개선.
